# 训练项目本体完备性全面迭代计划（2026-05-12）

## 1. 当前目标

本阶段的最高优先级不是 Git、发布、日报、报告页美化，也不是继续增加游戏数量，而是把现有训练项目本体做扎实。

“做扎实”的含义不是页面更热闹，而是每个训练模块都能回答四个问题：

1. 它到底训练什么能力。
2. 它的任务机制是否真的刺激这个能力。
3. 它是否能根据用户表现调整难度，而不是固定题面循环。
4. 它是否能留下足够数据，让用户知道下一轮怎么练。

因此，后续迭代的主线是：从“可运行 demo”推进到“范式明确、内容可变、难度可调、错误可解释、长期可复盘”的训练工具。

## 2. 第一性原则

### 2.1 有效训练的最低结构

每个训练项目至少应具备以下结构：

| 层级 | 要求 | 目的 |
|---|---|---|
| 训练目标 | 明确对应的认知能力和边界 | 避免把任何小游戏都包装成泛化训练 |
| 刺激机制 | 题目、刺激、奖励或干扰项围绕目标能力设计 | 保证训练压力打在正确位置 |
| 参数化内容 | 题目可生成、可变体、可记录 seed 或 contentVersion | 防止固定题库记忆化 |
| 难度调节 | 根据正确率、反应时、span、错误类型等调整下一轮 | 让训练停留在能力边界附近 |
| trial 数据 | 保存条件、刺激、反应、正确性、反应时、错误类型 | 支撑复盘和长期趋势 |
| 诊断反馈 | 输出主要错误模式和下一轮训练建议 | 让用户知道如何改进 |
| 口径约束 | 不宣称临床诊断、治疗或通用智力提升 | 保持产品边界可信 |

### 2.2 成熟度分级

后续判断模块优先级时，不再只看“有没有页面”，而按训练成熟度分级。

| 等级 | 名称 | 判断标准 |
|---|---|---|
| L0 | 静态 demo | 页面能打开，但内容固定、指标弱、训练目标不清 |
| L1 | 可完成训练 | 有完整交互、结果保存、基础反馈 |
| L2 | 参数化训练 | 有 seed/contentVersion，题目或刺激可变，并保存 trial 明细 |
| L3 | 范式指标训练 | 能输出对应范式的核心指标和错误类型 |
| L4 | 自适应训练 | 难度能随表现变化，并能给出下一轮处方 |
| L5 | 长期训练闭环 | 多日趋势、内容版本、训练计划和弱项追踪稳定联动 |

当前仓库已经大体越过“能不能保存结果”的阶段。主要矛盾转为：不同模块的 L2-L4 完成度不均衡，部分项目仍接近固定 demo。

## 3. 当前状态判断

### 3.1 已进入较强训练化的模块

这些模块已有参数化、自适应、错误分型或关键范式指标基础，后续以验证、微调和长期趋势联动为主。

| 系统 | 模块 | 当前判断 |
|---|---|---|
| 注意力 | Schulte、Focus、CPT、Stroop、Flanker | 已从静态交互推进到条件、反应时、错误类型和反馈闭环 |
| 工作记忆 | N-Back、Corsi、Digit Span | 已有负荷调整、span 或敏感性指标基础 |
| 执行功能 | Stop Signal、Go/No-Go、Task Switching | 已有抑制/切换指标与自适应基础，但 Stop Signal 仍需继续校准 SSRT 口径 |
| 推理/空间 | Raven、Mental Rotation | 已有模板化或条件化指标基础 |
| 决策/风险 | BART、Iowa、Hanoi | 已开始记录风险、收益、路径或规划偏离信息 |
| 概率/元认知 | Monty Hall、Gambler Fallacy、Confidence Judgment、Base Rate、Bayes Update | 已完成部分参数化和校准指标基础 |
| 执行学习 | WCST、Reversal Learning | 已具备规则转换、持续错误或反转学习指标基础 |

这些模块下一步不应被反复“重做一遍”，而应通过通关抽测和报告/计划联动确认训练信息真的可读、可用。

## 4. 后续批次计划

### 4.1 第一批：规划与社会决策

第一批应优先做 `london-tower`、`sliding-puzzle`、`ultimatum-game`、`trust-game`、`prisoner-dilemma`。

选择原因：

1. 这些模块多为封闭式任务，结果可客观评分，适合先把训练本体做深。
2. 当前短板集中在“任务能玩，但训练诊断不够强”。
3. 做完后，规划能力和社会决策两条能力域会从 demo 级别提升到可复盘训练级别。

验收标准：

| 模块 | 必须补齐 |
|---|---|
| London Tower | 题目 id、最少步数、首步延迟、实际步数、非法移动、回退、最优偏离、下一题难度处方 |
| Sliding Puzzle | 可解局面生成、难度层级、最短路径估计或启发式距离、实际步数、重复状态、回退循环、完成效率 |
| Ultimatum Game | offer ratio、角色、接受/拒绝、最低接受阈值、公平敏感性、反应时、不同报价区间表现 |
| Trust Game | 每轮投资、回报、伙伴策略、信任变化、互惠反应、背叛后恢复、策略说明 |
| Prisoner Dilemma | 双方动作、收益、对手策略标签、合作率、报复/宽恕、tit-for-tat 适应、末轮效应 |

第一批不处理报告页大改，不处理 Daily Plan 大改。只允许在必要时补最小 smoke 字段，以证明训练结果能落库。

### 4.2 第二批：创造力任务

第二批处理 `alternative-uses`、`remote-associates`、`torrance-creative`。

这一批的关键不是“自动打高分”，而是建立可信评分边界。

验收标准：

| 模块 | 必须补齐 |
|---|---|
| Alternative Uses | 题目变体、答案数量、类别数、重复/空泛答案标记、fluency/flexibility/elaboration，originality 只做透明启发式或留给人工评分 |
| Remote Associates | 题库扩充、难度分级、正确答案、提示层级、反应时、放弃率、线索距离说明 |
| Torrance Creative | 图形/提示池、标题/细节/类别/变形指标、原始答案保存边界、非标准化声明 |

创造力模块不允许把文本长度、答案数量或随机权重包装成“创造力分数”。可以给练习反馈，但必须说明是训练反馈，不是标准心理测评。

### 4.3 第三批：社会认知

第三批处理 `sally-anne`、`eyes-reading`。

验收标准：

| 模块 | 必须补齐 |
|---|---|
| Sally-Anne | 场景池、第一阶/第二阶信念、现实控制题、记忆控制题、错误类型、解释反馈 |
| Eyes Reading | 合规材料来源、题项 id、选项、正确性、情绪类别、易混类别、词汇理解提示 |

这批必须特别控制宣传口径。它们只能作为社会认知练习，不做诊断、不暗示筛查结果。

### 4.4 第四批：已强化模块二次校准

当前已经强化过的模块进入第二轮校准，而不是继续大拆大改。

重点包括：

1. Stop Signal：复核 SSD staircase、stop 成功率、SSRT 估算、go waiting 检测。
2. BART：复核 adjusted average pumps、爆炸率、学习曲线和风险反馈。
3. Confidence Judgment：补 calibration curve、overconfidence、Brier score 或分箱准确率。
4. Raven：检查模板题是否可验证、是否存在明显答案泄漏。
5. N-Back/Corsi/Digit Span：验证自适应是否过快、过慢或容易震荡。

## 5. 多 agent 协同方案

### 5.1 主 agent 职责

主 agent 不抢具体实现，但必须负责目标纠偏和闸门。

主 agent 每轮只做这些事：

1. 选择批次和模块。
2. 给每个子 agent 指定唯一写入范围。
3. 明确理论目标、必填指标和验收条件。
4. 检查子 agent 是否改错路径、越权改文档、越权改 Git 或扩散到无关模块。
5. 等所有 worker 完成后，再启动审计。
6. 根据审计结果决定返工、收口或提交。

### 5.2 Worker 闸门

每个 worker 必须遵守：

1. 先确认当前工作目录是本项目仓库根目录：`<repo-root>`。
2. 只读必要的模块文件、共享训练结果契约和相关本地文档。
3. 只修改分配到的 `.html` / `.js` 文件。
4. 不修改 Git、文档、README、package、报告页、Daily Plan，除非主 agent 明确授权。
5. 不新增临时脚本，不大范围重构共享文件。
6. 完成后至少运行 `node --check <module>.js`。
7. 回报必须包含：改动文件、训练目标映射、保存字段、验证命令、残余风险。

### 5.3 审计 agent 闸门

审计 agent 只在 worker 全部完成后启动，避免抢跑拿到中间状态。

审计内容：

1. `git diff --name-only` 是否只包含授权文件。
2. 每个模块是否真的新增了 trial/metrics/seed/contentVersion 或对应等价字段。
3. 训练指标是否对应理论目标，而不是只新增 UI 文案。
4. 是否出现固定题面、固定答案、固定节奏导致记忆化。
5. 是否出现医疗化、诊断化或夸大训练效果的文案。
6. `node --check`、`npm run smoke:pages`、`npm run smoke:flows`、`npm run check` 是否通过。

### 5.4 并行上限

最多同时启动 6 个子 agent，但不是每轮都必须用满。

建议：

| 场景 | 并行数 |
|---|---|
| 同一能力域、文件互不重叠 | 3-5 |
| 涉及共享脚本或结果契约 | 1-2 |
| 开放题评分边界不清 | 2-3 |
| 审计阶段 | worker 完成后再启动 1-2 个 |

并行的价值是减少等待，不是降低主 agent 的判断责任。

## 6. 每个模块的完成定义

一个训练模块只有同时满足以下条件，才算本体完成一轮：

1. 用户能从首页进入并完成训练。
2. 训练过程不是固定题面或固定节奏。
3. 至少有一个难度、内容、干扰、奖励或序列参数会变化。
4. 结果保存包含 summary 和 trial 级别信息。
5. summary 里至少有一个范式核心指标，而不是只有总分。
6. 错误能被分型，例如遗漏、误报、回退、冲动开局、持续错误、低信心正确、高信心错误。
7. 结束反馈能指出下一轮应该增加、降低或保持什么训练负荷。
8. 文案不承诺诊断、治疗或泛化智力提升。
9. 静态和流程门禁通过。

## 7. 验证门禁

每个批次完成后，主 agent 应运行：

```powershell
npm run check
npm run smoke:pages
npm run smoke:flows
git diff --check
```

对单个模块，worker 至少运行：

```powershell
node --check <module>.js
```

如果本轮修改涉及 flow smoke 覆盖字段，应追加：

```powershell
node --check scripts/flow-smoke.js
npm run smoke:flows
```

`git diff --check` 在 Windows 上可能出现 LF/CRLF 提示，应区分换行提示和真实 whitespace error。

## 8. 文档与 Git 策略

文档和 Git 是辅助任务，不进入训练本体优先队列。

处理原则：

1. 只有当一批训练本体完成并通过门禁后，才写实施记录。
2. 只有当本地状态稳定后，才提交 Git。
3. 网络不好时不反复消耗精力在 push 上，按 `31-github-submit-runbook-20260512.md` 统一处理。
4. 文档链接必须使用相对路径或当前仓库公开入口，不再写入个人账号空间、旧 GitHub 空间或本机绝对路径。

## 9. 推荐下一轮执行顺序

下一轮建议按下面顺序推进：

1. 执行第一批 P0：`london-tower`、`sliding-puzzle`、`ultimatum-game`、`trust-game`、`prisoner-dilemma`。
2. 主 agent 先给每个 worker 一个单模块写集和验收字段。
3. worker 完成后，主 agent 检查写集是否越界。
4. 再启动审计 agent 做训练目标和数据字段复核。
5. 通过后运行项目门禁。
6. 只在门禁通过后写实施记录。
7. 最后视网络状态决定是否 Git 提交和推送。

如果第一批推进顺利，第二轮再进入创造力模块。原因是创造力评分最容易出现伪精确，必须在封闭任务的训练化模式稳定后再做。

## 10. 本计划的判断

当前项目不是缺页面，而是缺部分模块的训练深度。后续最重要的不是扩大功能面，而是让每个训练任务都能形成稳定的训练压力、可复盘的数据和可执行的下一轮处方。

因此，短期最有价值的路线是：

1. 先补强规划和社会决策。
2. 再谨慎处理创造力评分。
3. 再补社会认知材料和错误分型。
4. 最后让报告页和每日计划吃到这些更强的训练数据。

只有训练本体先变强，报告、计划、发布和 GitHub 才有实际承载价值。
