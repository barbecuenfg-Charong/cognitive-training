# 注意力系统级训练闭环与趋势反馈计划（2026-05-13）

## 1. 目标

本轮不再把注意力训练当成一组独立小游戏，而是把它们收束成一个可持续训练系统。

目标只有一个：让用户每次练完注意力模块后，能明确知道自己在什么能力上变强、什么能力还弱、下一轮该练什么，而不是只看到某一局分数。

## 2. 现状复核

当前注意力系统里，`Schulte`、`Flanker`、`Stroop`、`Focus`、`CPT`、`Go/No-Go`、`Stop Signal` 已经具备不同程度的：

- `seed`
- `contentVersion`
- `summary`
- `trials` 或 `trialLog`
- 错误分型
- 自适应或下一轮建议

这说明训练本体已经不再是纯 demo。

但主要矛盾也变得更清楚了：

1. 模块级数据很多，但系统级画像很少。
2. 每个模块都在说自己的话，缺少统一的注意力能力坐标系。
3. 报告页能看 session，但还不能稳定回答“这一类注意力训练到底应该怎么接着练”。
4. 真实浏览器闭环还没有作为默认验收入口。

## 3. 这轮要解决什么

### 3.1 系统级画像

把注意力模块统一映射到少数几个能力维度：

- 视觉搜索
- 选择性注意
- 持续警觉
- 冲突控制
- 反应抑制
- 稳定性与节律控制

每个模块都要能归入这些维度中的一到两个主维度，避免每个页面只说自己的指标，最后用户看不懂整体进展。

### 3.2 跨 session 趋势

报告页不能只展示“本次 session 结果”，还要能看：

- 最近 7 次
- 最近 14 次
- 最近 30 次

的趋势变化，至少包含：

- 准确率
- 反应时
- 干扰成本
- 误报 / 漏误
- 稳定性
- 下一轮建议是否在变强

### 3.3 训练处方聚合

现在每个模块已经能给出自己的下一轮建议，但系统层还缺一个聚合器：

- 今天优先练哪一类注意力
- 哪个模块应该降负荷
- 哪个模块可以提高难度
- 哪些模块适合轮换，哪些需要连续巩固

## 4. 推荐方案

### 方案 A：先做报告层聚合

做法：

- 不动训练逻辑，只新增系统层汇总。
- 在 `report.js` 里增加注意力系统区块。
- 从 `TrainingResults` 的 `summary` / `metrics` 中提取统一字段。

优点：

- 风险最低
- 能最快看到系统层效果
- 不会影响现有训练页

缺点：

- 依赖现有字段质量
- 不能立刻解决模块间字段不一致

### 方案 B：先统一字段，再做聚合

做法：

- 先给注意力模块补一层统一字段映射。
- 再做系统级 report 和趋势视图。

优点：

- 后续维护干净
- 聚合口径更稳

缺点：

- 改动面更大
- 周期更长

### 方案 C：并行做字段统一 + 报告层

做法：

- 按注意力系统划一个共享 helper。
- 训练页继续保留原字段，同时补 canonical 字段。
- 报告页同步读 canonical 字段和旧字段。

优点：

- 风险和收益平衡最好
- 既能尽快出效果，也能保证后续可维护

缺点：

- 需要控制写集，避免 worker 扩散

推荐选择：**方案 C**。

## 5. 计划拆解

### 5.1 第一步：定义注意力系统 canonical 字段

目标是让注意力模块输出可以被统一消费，不再靠 report 页猜字段名。

建议补充的统一口径：

- `domain`
- `coreMetric`
- `errorProfile`
- `loadLevel`
- `trendDirection`
- `nextRecommendation`
- `attentionProfile`

优先覆盖：

- `schulte`
- `flanker`
- `stroop`
- `focus`
- `cpt`
- `go-no-go`
- `stop-signal`

### 5.2 第二步：改造报告页为系统级视图

`report.js` 需要新增一个注意力系统区块，至少展示：

- 系统总览
- 各模块主指标
- 最近多次训练趋势
- 当前最弱环节
- 下一轮优先顺序

如果字段不足，就先用保守回退，不要编造不存在的指标。

### 5.3 第三步：补系统级趋势聚合

建议新增一个共享聚合层，职责只做一件事：

- 把多个 session 的注意力训练结果折叠成可读趋势

趋势输出应包含：

- 每个维度的变化方向
- 近几次是否稳定
- 是否存在某模块反复卡住
- 是否出现难度过快上调

### 5.4 第四步：补真实验收

静态 smoke 只能证明页面没断，不足以证明训练链路真的可用。

因此本轮验收应加一层：

1. 从首页进入注意力模块。
2. 完成一次训练。
3. 保存结果。
4. 打开 report 读取结果。
5. 核对系统级建议是否可见。

## 6. 文件边界

本轮如果进入实现，优先级建议如下：

- `report.js`
- `src/shared/training-results.js`
- `src/home/config.js`
- `scripts/flow-smoke.js`
- 注意力模块各自的 `js` 文件
- `doc/INDEX.md`

不建议本轮扩展到：

- 新增游戏
- 大改首页视觉
- 改发布包结构
- 改 GitHub 推送流程

## 7. 验收标准

完成这一轮后，至少应满足：

1. 注意力模块能被系统层统一识别。
2. 报告页能显示系统级趋势，而不是只有 session 列表。
3. 用户能看懂“下一轮先练什么”。
4. 训练闭环是可验证的，不只是文字描述。
5. 不引入新的隐私或敏感路径暴露。

## 8. 下一步建议

如果你认可这个计划，下一步就进入实现计划阶段，再把注意力系统拆成 2-3 个小批次：

1. canonical 字段统一
2. report 系统层聚合
3. 真实浏览器闭环验收

这样不会把所有注意力模块一次性重写，也不会让主 agent 丢掉闸门。
